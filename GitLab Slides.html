What is Git?

Git is a Version Control System

  • Open Source
  • Distributed (vs Centralized)
  • Records changes, maintains history of files
  • Great branching model that adapts to most workflows
  • Fast & reliable
  • Command line, native and web-based apps

https://git-scm.com

Distributed does not require a constant connection to a central server Centralized does (SVN, P4, VSS) Great branching EXTREMELY lightweight Branches are sparse; back end stores only files that are changed on the branch vs. containing EVERY file

Git Internals

Object Types

  • Commit - Author, message, pointer to a tree of changes
  • Tree - Pointer(s) to file names, content, other trees
  • Blob - Data (source code, pictures, videos, etc.)

Git Internals (con't)

  • Stored as objects on the filesystem
  • Tags and Branches
    • Pointers to commits (lightweight)
    • Not full copies
    • Allows a name for a commit
      • Contains additional meta information

BLOB - BInary Large OBject Git compresses this Its Git’s way of storing file content (like a gzip)

Tag - can refer to a release, have release notes, etc.

Git is a

"Database of References"

AKA: Git is one giant graph!

(directed acyclic graph, to be specific )

By understanding the git data model, the overlying concepts should become more clear and less mystical.

  • Commits all reference their parent
  • Branches are named pointers to commits
  • ‘master’ is the canonical mainline branch
  • HEAD is a special pointer to the latest commit
    • Moves automatically (for current, active branch)

Performing a clone/checkout starts at the current commit and works up the parent chain, fetching tree+blob content!

Revert & Unstage

  • To remove files from staging, use git reset
    git reset HEAD <file>
  • This will unstage the file but maintain the modifications. To revert the file back to the state it was in before the changes (after ‘git reset’)
    git checkout -- <file>

Remember that staging is letting git know you want to commit the file. HEAD -- the last commit you pulled -- non-destructive Checkout -- DESTRUCTIVE Using --cache to keep the file in case you want to add it to .gitignore Files are removed with ‘git reset’ - any file(s) that are staged via ‘git add’ “HEAD .” -- all

  • To remove a file from disk and the repo use git rm
git rm <file>
git rm -r <dirname>   (recursive -- for directories)
  • To remove a file from the repo, but keep it on disk
git rm <file> --cache

Undo Commits

  • Add files and change commit message
    git commit --amend -m ”New message”
  • Undo the last commit and put everything back into staging
    git reset --soft HEAD^
  • Undo the last commit and remove changes
    git reset --hard HEAD^
  • Same as last, but going two commits back
    git reset --hard HEAD^^

Resetting after pushing can cause downstream issues In case of Geo replication In case another user pulls your changes before you reset Etc. ZZZ what exactly are these ‘issues’? Geo gets out of sync because you’ve now reset the branch after Geo has replicated everything Other users get out of sync because they’ve pulled or cloned between your push and your reset Essentially causes lots of potential conflicts that will need to be resolved

Cleaning Up Local Branches

Shows which branches may be “stale” (merged into master)

git remote prune origin

Cleans up the “stale” external references to those branches

git branch -a

Shows all branches

git branch -d branchname

To delete local branches that have been pushed to GitLab and then merged into master

Just be careful to only delete those branches that were stale and pruned.
